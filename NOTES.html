In here there are a few top level sections of the yaml file.

name: this defined the name for this action.
on: is the set of triggers that cause this action to run, there are many that are options, and the full list of those can be found here. The one's I used in all the workflows I configured are pull_request which happens whenever there is a change to a pull request and push which occurs when code is pushed, both have two sub fields, branches which limits which branches this will run on, and paths which say only run this when that certain file changes.

Jobs

For most of our services we have just three steps, and they do a pretty good job of explaining some of the overall things you need to be concerned about.

    - name: 'Update Repository'
      run: |
        git fetch origin ${{ github.sha }}
        git reset --hard ${{ github.sha }}

This step is the first one that runs, since we set the working-directory: 'C:\workspace\' in the defaults this assumes its in the correct github repo which has already been cloned once to that location. As soon as the action runner starts it will then execute all the commands in the run: | section.

git fetch origin ${{ github.sha }} will check the origin, the remote server we have named in our repo as origin for us this is github, for any changes to to the specific github commit/sha this build was requested for, in git/github the sha/sha256 is a hash of the state of the repo used to uniquely identify a commit in the history.

Then git reset --hard ${{ github.sha }} will reset the local state to match that commit, the --hard flag means it will delete or revert any local changes to the branch state.

This is a slightly more complex version then is really needed since we have a large repo and cloning takes a while. Since we trigger ~90 builds off the same source version this saves time rather then redownloading the repo each time.

    - name: 'Build'
      run: |
        sig tool msbuild '/t:clean;build' '/p:Configuration=Release' '/p:Platform=Any CPU' '/verbosity:minimal' '.\Dev\Corp\App\Service\CorpService\Signicast.Corp.App.Service.CorpService.sln'

This is the actual build set, like the first one it will run all the steps in the run: | section.

This script uses sig tool msbuild as a shortcut to find the local machines installed version of msbuild, this can be changed in the future to a hard path, or even to just msbuild as long as the exe is added to the machines PATH environment variable.

The actual parameters used in here are specific to msbuild, more info on that here, but simply this just runs the release build for the sln passed in at the end.

/t:clean;build: will run the clean and build steps of the sln.
/p:Configuration=Release: sets it to release mode
/p:Platform=Any CPU: will build for the configured Any CPU mode, if we have slns with other configured names this might need to be changed from the default.
/verbosity:mimimal: will print out any warnings/errors but not each action that the build is doing, since the logs are stored on github for free for I belive a retention period of 30 days or longer we could have full diagdiagdiagnostic set to see more info.

Again anything in ${{ something? }} is using template from the info/context of this action. so ${{ github.workflow }} is replaced with the name of the workflow, and ${{ github.sha }} is the unique hash of the commit so we always know what version of the source a release was made for.

This has a new field verses the others. The uses directive basically says rather then writing my own action I'm going to use this community/github defined action. The name of that goes right to a github action so actions/upload-artifact@v2 goes to the https://github.com/actions/upload-artifact repo, this is where all the info for this action is stored and updates/anything can be checked here including the entire source code that will be run. Any shared actions will have their code saved in here.

The other new parameter with is a set of common settings for that shared action. In this case.

name: is the name to use for the artifact that gets uploaded
path: is the paths to include. Since I didn't have a perfect way to find the artifact that will get uploaded, or if a build would have multiple like the replication/storage service, the script that generates these actions includes any .csproj file that has Installer in its name and builds the path accordingly. Once this step completes that artifact is not published onto github and ready to be downloaded. This could be replaced in the future with just a copy to a network location.

gen-github-actions

The source code is located at S:\Utilities\go\gen-github-action also built in go.

Usage

Usage:
  single <solution>                   : build a single solution
  list   <solution-list> <output dir> : runs on all solutions in the list

Usage of gen-github-action:  
  -c, --configuration string   Configuration to build (default "Release")
  -p, --platform string        Platform to build (default "Any CPU")
  -t, --targets string         Targets to build (default "clean;build")
      --trim string            Trim prefix
  -v, --verbosity string       Verbosity level (default "minimal")
  -w, --working-dir string     Working directory (default "C:\\workspace\\")

To rebuild the entire set of builds I used the list option.

$ gen-github-action list ./solutions.txt S:\git\sig\.github\workflows --trim 's:\git\sig' --working-dir 'C:\workspace'

This option will go through each line in the 'solutions.txt' file and assuming its a solution read it and all its dependent projects, using their path to construct the paths filter and then write the file out to the <output dir> that is the second argument used in execution. This will overwrite any files there without concern so if you make custom changes you might want to exclude them from the list.

Inside the code all of the functions besides run are argument parsing and other stuff, not really important to change the actual run you should change the lines in this. block here in that step. the code is annotated in both places.

// run will output the correct action into w, or anything that can be written to, like stdout
// or a specific file that was passed in
func run(w io.Writer, sln string, o options) error {
	if sln == "" {
		return fmt.Errorf("sln is required")
	}

  // reads in the file and converts the contents to UTF8 if they are not already
  // for some reason we found that some of the older slns/were in some weird UTF-16/le format
  // which was causing the regex to not work correctly, aka dam windows
	data, err := utfutil.ReadFile(sln, utfutil.UTF8)
	if err != nil {
		return fmt.Errorf("failed to read sln: %s", err)
	}

  // this converts the path the slns directory to be absolute, letting this method run with relative paths
	basedir, err := filepath.Abs(filepath.Dir(sln))
	if err != nil {
		return fmt.Errorf("failed to get absolute path of %s: %s", sln, err)
	}

	pathsArr := []string{}
	installerpaths := []string{}

  // now for each submatch in the 'projectRegex' we evaluate the path found in that reference to 
  // be relative from the base directory of the projects sln file
	for _, m := range projectRegex.FindAllSubmatch(data, -1) {
		realpath, err := filepath.Abs(filepath.Join(basedir, string(m[2])))
		if err != nil {
			return fmt.Errorf("failed to get absolute path of %s: %s", string(m[2]), err)
		}
    // if the project reference is a wix project we will also add it to a list of installer paths
		if strings.Contains(string(m[2]), ".wixproj") {
			installerpaths = append(installerpaths, filepath.ToSlash(fmt.Sprintf("%v/bin/%v/*.msi", strings.TrimPrefix(filepath.Dir(realpath), o.trim), o.configuration)))
		}

    // next step is to add it the the list of paths using the directorys path without the prefix and the 
    // dir name where that .csproj/.wixproj is found in
		pathsArr = append(pathsArr, fmt.Sprintf("      - '%v/**/*'\n", filepath.ToSlash(strings.TrimPrefix(filepath.Dir(realpath), o.trim))))
	}
	var (
		paths  = ""
		unique = map[string]struct{}{}
	)
	sort.Strings(pathsArr)

  // after all the paths have been added it goes through them to ensure there is only one unique entry 
  // for each path
	for _, p := range pathsArr {
		if _, ok := unique[p]; !ok {
			paths += p
			unique[p] = struct{}{}
		}
	}

	name := strings.TrimSuffix(filepath.Base(sln), filepath.Ext(sln))

  // start of the actual yaml definition here, all these fprintlns are careful to use only spaces
  // never tabs
	fmt.Fprintf(w, "// This file is generated by `gen-github-actions --sln '%v'`\n", sln)
	fmt.Fprintf(w, "name: '%v'\n\n", name)
	fmt.Fprintln(w, "on:")
	fmt.Fprintln(w, "  push:")
	fmt.Fprintln(w, "    branches: [main]")
	fmt.Fprintln(w, "    paths:")
	fmt.Fprintln(w, paths)
	fmt.Fprintln(w, "  pull_request:")
	fmt.Fprintln(w, "    branches: [main]")
	fmt.Fprintln(w, "    paths:")
	fmt.Fprintln(w, paths)

	// if name == "Signicast.Corp.SwitchboardGui" {
	// 	fmt.Fprintln(w, "  workflow_dispatch:")
	// }

	fmt.Fprintf(w, `
defaults:
  run:
    working-directory: '%v'

jobs:
  build:
    runs-on: self-hosted

    steps:
    - name: 'Update Repository'
      run: |
        git fetch origin ${{ github.sha }}
        git reset --hard ${{ github.sha }}
`, o.workingDir)

  // if the name of the build is SwitchboardGui then it will do a completely separate definition
	if name == "Signicast.Corp.SwitchboardGui" {
		sb := filepath.Join(o.workingDir, "SwitchboardImage")
		fmt.Fprintf(w, `
    - name: 'Clean'
      run: |
        Remove-Item -Path '%v' -Recurse -Force -ErrorAction Continue -WarningAction Continue
`, sb)

		fmt.Fprintf(w, `
    - name: 'Build'
      run: |
        sig build --tags gui --metadata -c Release --skip-recipts --switchboard %v --replace-sig %v
`, sb, o.workingDir)

    // this step copys the output from switchboard to `SwitchboardCiImages` on sigcode right now
    // this can be changed to something else when this becomes the production/real way to do it
		fmt.Fprintf(w, `
    - name: 'Copy Switchboard Image'
      run: |
        Copy-Item '%v' -Destination '\\sigcode\Drops\SwitchboardCiImages\${{ github.workflow }}-${{ github.sha }}-SwitchboardImage' -Recurse -Force -ErrorAction Continue -WarningAction Continue
`, sb)

  // otherwise it simply adds a single msbuild step into the routing
	} else {
		fmt.Fprintf(w, `
    - name: 'Build'
      run: |
        sig tool msbuild '/t:%v' '/p:Configuration=%v' '/p:Platform=%v' '/verbosity:%v' '.\%v'
`, o.targets, o.configuration, o.platform, o.verbosity, strings.TrimPrefix(sln, o.trim))
	}

  // if there were any .wixprojs included in the sln this will add a step to copy any msis from them
	if len(installerpaths) > 0 {
		fmt.Fprintln(w, `
    - name: 'Upload Artifacts'
      uses: actions/upload-artifact@v2
      with:
        name: '${{github.workflow}}-${{github.event.sha}}-installer'
        path: |`)
		for _, installerpath := range installerpaths {
			fmt.Fprintf(w, "          %v\n", filepath.ToSlash(filepath.Join(o.workingDir, installerpath)))
		}
	}

	return nil
}

3. Dip

Program Editor

A dip program consists of a few global settings and then a list of dips. Essentially the hierarchy looks something like this.

Dip Program
- Dips
  - Main Steps*
    - Sub Steps*
      - Operations*

*these can have operation parameters

Since rules can change at any time, whenever a dip program is loaded into the editor it goes through all the levels of main/sub/operation to update the rule backing it. This rule contains info on the name, the type of parameter like list/number/on-off/pot number. If a rule no longer exists in the set it will become a invalid operation, this only affects the editor and validation, the controller itself makes no effort to validate a program before downloading.

Inside the controller it maintains a list of Active Dip Programs and whenever one is ready to be downloaded it will go to the current most active program, unless the number of dip operations is different.

Each Dip Program starts out a Wash Program chosen and maintained separately, in mod-7 there is a number of different configurations that can be chosen for each step but they must always progress through like this.

Wash
DC 1
- dry tunnel 1
DC 2
- dry tunnel 2
DC 3
- dip cranes
DC 4-5  // can be multiple, always intermediate dip
- dip cranes
DC 6-7  // can be multiple, must end with a seal dip
- final dry cranes

In the lower mods its simple, their is only one configuration for the plant and the molds go in and out of the dry tunnel or inter-coat as the call it down there, as in between coats of the dipping juice.

Slurry Pot's

Ok this is stored in two separate places for right now, yes its annoying but it has never been a huge priority to fix and would require a migration of at least one table maybe two if we decide to go to a completely new schema.

For the Dip Program Editor: This is tied to the Dip Cell Config. Basically you change the assignment for the dip cell/pot you just change the DIP_SLURRY_ASSIGNMENT table to the new DIP_SLURRY_TYPE and the DIP_CELL_CONFIG table. Here's a query that shows the relationship.

select assignment.DIP_SLURRY_ASSIGNMENT_ID
    , assignment.SLURRY_POT_NO
    , slurry.[DESCRIPTION]
    , config.CELL_NAME
    , config.CELL_DESCRIPTION
from dbo.DIP_SLURRY_ASSIGNMENT assignment
inner join dbo.DIP_SLURRY_TYPE slurry on slurry.DIP_SLURRY_TYPE_ID = assignment.DIP_SLURRY_TYPE_ID
inner join dbo.DIP_CELL_CONFIG config on config.DIP_CELL_CONFIG_ID = assignment.DIP_CELL_CONFIG_ID

For the Timeline/Dip Slurry Recipe: This is a little more straight forward. In Cell controller for each dip cell/station their is a name of the recipe in each pot. To add more pots you need to modify the configuration in the actual controller. To change the names at the station you can use the db maintenance form in switchboard, or just edit the cell controller values itself.

The recipes for this are saved/edited using the Dip Materials entry in the tree.

Adding and modifying the dip rules

Each rule is tied to a specific DIP_CELL_CONFIG in the DIP_RULE. There are two types of rules, regular dip rules, and wash rules. In order to edit these the best way is to use the editor on a old machine with access 2014 or its excel interop drivers installed.

Download the sheet, make your changes and re-upload. This has only been done once that I know about and if it doesn't work you'll have to edit the excel manually.

Salesforce

API Surface

The main api that we use currently is called SOQL, or Salesforce-Object-Query-Language. Allows us to define and run any custom query's against salesforce inner object model and any custom fields we have defined for signicast/forms.

The credentials for this are stored inside the configuration service. Any account should be added to salesforce with the help of 'Gia Capriotti gcapriotti@formtechnologies.com' from forms.

The query itself is pretty easy.

GET https://formtechnologies.my.salesforce.com/services/data/v37.0/query?q=<query>

The query is just sql specific to salesforce, for example to get the of signicast accounts we use this query. And then use the [SerializeAs(Name="Whatever")] to control the output coming back to us.

			var q = StripWhitespace($@"
				SELECT 
					Name
					, Description
					, Type
					, ParentId
					, BillingAddress
					, ShippingAddress
					, Phone
					, Id
					, AnnualRevenue
					, Website
					, CreatedDate
					, LastModifiedDate 
				FROM Account
				{(options?.LastModifiedDate is DateTime modified ? $"{GetWhereOrAnd()} LastModifiedDate		>= {modified:yyyy-MM-dd'T'hh:mm:ss'z'}" : "")}
				{(options?.CreatedDate is DateTime created       ? $"{GetWhereOrAnd()} CreatedDate			>= {created:yyyy-MM-dd'T'hh:mm:ss'z'}" : "")}
				{(!string.IsNullOrEmpty(options?.Type)           ? $"{GetWhereOrAnd()} Type                 =  '{options.Type}'" : "")}
				{(options?.IsSignicastAccount is bool sigAccount ? $"{GetWhereOrAnd()} Signicast_Account__c =   {sigAccount}" : "")}
			");

Updating a object back is as simple as posting the same object used to read it back to the api whatever fields that are present will be applied to the object. For example

POST https://formtechnologies.my.salesforce.com/services/data/v20.0/sobjects/Opportunity/{Id}

{
	"IC_Opportunity_Number__c": 1231231,
	"Signicast_Estimator__c", 5578, 
}

So when <Id> is set the the id of actual Opportunity it will update those two fields without setting anything else. And thats kinda the only thing we do.

Above is the definition for the SignicastGoalSheetLister this includes several different types.

CachedPlantGoalSheetLister: Is the most basic unit, this will load data from various different report caches and then store the result in the IGoalSheetStore.
ManualGoalsObjectivesLister: Will load data from the manual goals entry form, this is not really being used. Also stores its data into the IGoalSheetStore
RollupPlantGoalSheetLister: The first parameter is the entity that this lister represents, the other variadic parameters are IPlantGoalSheetListers each one of these is executed and the values in its goalsheets are added together to create a new one which's values are the sum of all its components.
GraphGoalSheetLister: Does not gather data itself, instead pulls its goalsheets out of the IGoalSheetStore
SerialGoalSheetLister: Runs through all the IPlantGoalSheetListers and executes them returning the goalsheet from the store that matches the CorporateEntity in the first parameter spot.
